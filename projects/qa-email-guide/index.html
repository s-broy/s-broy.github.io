superscript
sup
rules

In Umbraco, disclaimer codes appear superscripted and within brackets; for example: [LEGALTERM].

The disclaimer code is a disclaimer that was created in Umbraco with a specific name, usually labelled something related to what they are disclaiming.

Umbraco automatically replaces the disclaimer code in brackets with numbers in the preview link and in emails. Thus, Umbraco automatically numbers them based on the order they appear in. A disclaimer code that appears more than once in the email gets the same number as when it first appeared in the email.

They have various rules, so click on it to read more about it.

This is a module image.

In Umbraco, most modules can get assigned an image and a CTA. An image has a link, an alias and an ALT text field. A CTA has an alias and a link field.

Single quotes use the straight apostrophe HTML code. Double quotes use the straight quotation mark code. None of these should ever be curly or smart. If they are, recommend writing them out using their respective HTML code or entity.

Trademark, registered, and copyright signs never go in <sup>, because the proprietary font automatically superscripts them. The proprietary font isn't supported within email clients, only in Web view, so these signs appear larger within email clients.

These signs also never go in <strong> tags, because they make them difficult to read, especially on mobile devices.

There are a few "hacks" we currently use to avoid a line breaking and starting a new line, but because these pieces of code essentially abuse how different browsers interpret CSS, we try to avoid using them. This is how the code snippets would look like in the "View page source" file (note that the Chrome's or Firefox's inspector do not display these code snippets):

  • <span style="white-space: nowrap">This line doesn't break</span>
  • Land&#160;Cruiser doesn't break
  • Superscript&#8204;2 doesn't break

Also note when considering padding, width and height for mobile devices and apps display these slightly different based on their own interface. Also, adjusting a body copy or element's padding and/or line break on one mobile device might make that same body copy or element look strange on another device with a larger or smaller screen.
